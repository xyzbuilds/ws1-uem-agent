# ws1 v0.1.0 — Initial PoC release

**Status:** proof-of-concept · not for production use yet

`ws1` is an agent-shaped CLI for Workspace ONE UEM. It lets a skill-capable
agent (or a human) drive WS1 operations from natural-language goals,
with a finite envelope contract on stdout and a browser-approval flow
in front of every destructive op.

This is the first tagged release — feedback welcome before we cut v0.2.

---

## What's in the box

- **`ws1 setup`** — interactive first-run wizard. Walks through tenant
  hostname, region, OAuth client, and default Org Group. Re-running is
  the reconfigure path; existing values are offered as defaults.
- **`ws1 setup --advanced`** — configure multiple profiles (`ro` /
  `operator` / `admin`) in one run.
- **`ws1 status`** — single-envelope snapshot of current config (active
  profile, configured profiles, tenant, region, OG, last audit seq).
- **`ws1 doctor`** — connectivity + auth + OG sanity-check, with a
  colored TTY checklist alongside the JSON envelope.
- **~980 operations** across `mdmv1` … `systemv2`, auto-registered from
  the WS1 OpenAPI specs. Filterable via `ws1 ops list --section / --tag
  / --class / --filter / --summary` and `ws1 ops search <pattern>`.
- **Browser approval flow** for every `destructive`-class op (and `write`
  ops above a configurable blast-radius threshold). The CLI binds an
  ephemeral `127.0.0.1:<random>` server, opens a localhost approval URL
  in the user's default browser, and proceeds only after click. The
  agent never sees the request id.
- **Hash-chained JSONL audit log** at `~/.config/ws1/audit.log`,
  verifiable via `ws1 audit verify`.
- **OS keychain** (macOS / Windows DPAPI / Linux secret-service) for
  OAuth client secrets. Opt-in disk fallback (`WS1_ALLOW_DISK_SECRETS=1`)
  for CI / unattended environments — see `SECURITY.md`.
- **Companion `ws1-uem` skill** under `skills/ws1-uem/` for Claude-family
  runtimes (SKILL.md ~3-4k tokens, plus concept + pattern docs).

## Install

Pick the binary that matches your machine. The default install drops it
in `~/.local/bin`, which is on `PATH` by default on most modern shells
— **no sudo required**.

### macOS — Apple Silicon (M1/M2/M3/M4)

```bash
mkdir -p ~/.local/bin
curl -L https://github.com/xyzbuilds/ws1-uem-agent/releases/download/v0.1.0/ws1-darwin-arm64 \
  -o ~/.local/bin/ws1 && chmod +x ~/.local/bin/ws1
```

### macOS — Intel

```bash
mkdir -p ~/.local/bin
curl -L https://github.com/xyzbuilds/ws1-uem-agent/releases/download/v0.1.0/ws1-darwin-amd64 \
  -o ~/.local/bin/ws1 && chmod +x ~/.local/bin/ws1
```

### Linux — x86_64

```bash
mkdir -p ~/.local/bin
curl -L https://github.com/xyzbuilds/ws1-uem-agent/releases/download/v0.1.0/ws1-linux-amd64 \
  -o ~/.local/bin/ws1 && chmod +x ~/.local/bin/ws1
```

### Linux — arm64

```bash
mkdir -p ~/.local/bin
curl -L https://github.com/xyzbuilds/ws1-uem-agent/releases/download/v0.1.0/ws1-linux-arm64 \
  -o ~/.local/bin/ws1 && chmod +x ~/.local/bin/ws1
```

### If `~/.local/bin` isn't on your `PATH`

Add it once and reload your shell:

```bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc
```

### Windows

Download `ws1-windows-amd64.exe` from the release page, rename to
`ws1.exe`, and drop it in a folder already on `%PATH%` (e.g.
`C:\Users\<you>\AppData\Local\Programs\ws1\`).

### Want it system-wide instead?

Install to `/usr/local/bin` so every user on the machine can run `ws1`
(needs `sudo` on macOS):

```bash
sudo curl -L https://github.com/xyzbuilds/ws1-uem-agent/releases/download/v0.1.0/ws1-darwin-arm64 \
  -o /usr/local/bin/ws1 && sudo chmod +x /usr/local/bin/ws1
```

### Verify

```bash
ws1
```

You should see a teal mascot + "ws1 CLI" + "Workspace ONE UEM —
agent-first CLI" + a yellow "No configuration found. Run `ws1 setup`."

## First-run setup (live tenant)

Before running `ws1 setup`, provision an OAuth client in your tenant:

> WS1 console > Groups & Settings > Configurations > **OAuth Client
> Management** > **Add**
>
> Assign a console role that matches the profile you'll configure
> (`ro` → Read-Only, `operator` → Console Administrator, `admin` →
> elevated). The role enforcement happens server-side; the CLI gates
> are belt-and-braces.

Then:

```bash
ws1 setup
```

You'll be asked for:

1. **Tenant hostname** — use the `as`-prefix REST API URL (e.g.
   `as1784.awmdm.com`), not the `cn`-prefix console URL.
2. **Region** — `uat` / `na` / `emea` / `apac`.
3. **OAuth client_id** + **client_secret**.
4. **Default Org Group** — fetched from your tenant; pick the one most
   of your work targets.

Setup runs an OAuth round-trip + a smoke-test API call, then writes a
"Setup complete" summary with your next-steps cheat sheet.

## Known limitations / not in v0.1.0

- **Headless approval is deferred to v2.** v0.1 requires a browser on
  the machine running the CLI for any destructive op.
- **Active-profile state is OS-user-scoped, not shell-scoped.** Switching
  to `operator` affects every process running as your user. See
  `SECURITY.md` for v2 mitigations.
- **Audit log writable by the agent.** v0.1 hash-chains entries so
  tampering is *detectable* via `ws1 audit verify`, not *prevented*.
- **No long-running daemon yet.** Every CLI invocation is independent;
  approval state doesn't persist across processes.
- **Mock-mode OAuth `Sync` flag** — the v1 vs v2 OG-search response keys
  are decoded together (`LocationGroups` + `OrganizationGroups`), but
  some op response shapes haven't been hand-validated against a real
  tenant yet. File issues for any divergence.

## Demo path (suggested for stakeholder calls)

For a leadership demo, the highest-impact 90 seconds:

```bash
ws1 setup                                           # one-time: 4 prompts
ws1 status                                          # confirm config
ws1 ops search wipe                                 # find ~5 destructive ops
ws1 mdmv1 devices search --pagesize 5               # read-class, fast
ws1 mdmv1 devices lock --id <real-device-id>        # write-class — triggers
                                                    # browser approval
```

The lock command is reversible (`ws1 mdmv1 devices unlock --id ...`)
so it's a safe demo target.
